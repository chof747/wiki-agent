# Integration Harness

This document defines the repo-owned development and integration-test harness for the Wiki-Go **Comment Agent**.

Use it when implementing or extending local runtime setup, integration tests, or future CI procedures that exercise the real Wiki-Go helper-command boundary.

## Goals

- Keep `wiki-agent` running on the host during development.
- Keep Wiki-Go isolated in a repo-owned disposable container.
- Reuse a developer-managed Postgres service locally without letting this repo manage that shared service.
- Preserve the real `wikigo-*` subprocess contract used by the **Scanner** and later the **Runner**.
- Make the same harness callable locally and in CI.

## Non-Goals

- Replacing the current fast default local test baseline of `uv run pytest`.
- Baking developer-global shell configuration or machine-global Wiki-Go CLI installs into the workflow.
- Treating Postgres as repo-owned infrastructure in local development.

## Topology

### Local development

- `wiki-agent` runs on the host.
- Postgres runs outside this repo as a developer-managed service.
- Wiki-Go runs in a repo-owned disposable container.
- The harness creates repo-local `wikigo-*` shim commands and injects them into `PATH` only for harness-managed commands.

### CI

- The same harness entrypoint is reused.
- Wiki-Go runs in the same repo-owned disposable container model.
- CI may use a disposable Postgres service owned by the CI job.

## Harness Contract

The repo should expose one small Python entrypoint with subcommands for lifecycle operations such as:

- `up`
- `reset`
- `test`
- `down`

The first concrete entrypoint lives at:

```bash
env UV_CACHE_DIR=/private/tmp/uv-cache uv run wiki-agent-integration <subcommand>
```

The harness owns:

- Wiki-Go container lifecycle
- generated repo-local wrapper config
- generated repo-local `wiki-agent` runtime config
- creation of repo-local `wikigo-*` shim commands
- scenario seeding and reset
- per-run Postgres database setup and teardown

## Wiki-Go Boundary

- The harness preserves the exact helper-command surface expected by the application, including `wikigo-comments-scan`.
- The helper surface should be defined up front for the expected Wiki-Go operations, even if some commands are initially unused.
- Wrapper scripts read from a generated repo-local config file, not developer-global shell state.
- The generated runtime wrapper config should be ignored by Git. A committed template may define defaults.

## Wiki-Go Runtime Rules

- Pin a specific Wiki-Go image version by default.
- Do not track a floating image tag such as `latest` by default.
- Let the harness allocate or discover the localhost port and write it into generated repo-local config.
- Use fixed test credentials generated by the harness setup flow.
- Seed and reset Wiki-Go through its public interface and helper commands, not internal storage shortcuts.
- Reset logical Wiki-Go state per test case without requiring a full container restart.
- Use a unique per-test page namespace such as `/__tests__/<run-id>/<case-id>/...`.

## Postgres Runtime Rules

### Local development

- Treat Postgres as externally provided, even if that developer-managed service itself runs in a shared container.
- The harness must stay database-scoped and must not restart, wipe, or reconfigure the shared Postgres service.
- The harness uses an admin DSN for `CREATE DATABASE` and `DROP DATABASE`.
- The application under test uses a narrower generated runtime DSN for the per-run database.

### CI

- CI may supply both the admin DSN and runtime DSN from a disposable Postgres service.

## Fixtures

- Store committed Wiki-Go scenario fixtures in this repo.
- Prefer small per-scenario fixture sets over one large shared seed dataset.
- Apply fixtures during harness reset so the same scenarios are reusable in local development and CI.

## Verification Scope

### First slice

- Prove the real **Scanner** dry-run path against real Wiki-Go fixtures.
- Express harness-backed integration assertions as dedicated pytest cases under `tests/integration/`, executed through the harness-managed `test` command.
- Scaffold the harness structure so later issues can extend it for **Worker**, **Runner**, and Postgres-backed **Comment Job** flows.

### Future slices

Future runtime issues should extend this harness rather than create parallel setup flows for:

- durable **Scanner** enqueueing
- Postgres-backed **Comment Job** persistence
- **Worker** execution
- **Runner** subprocess integration
- end-to-end update and rejection flows

## Local Verification

- Keep `uv run pytest` as the default fast baseline for now.
- Run Wiki-Go and Postgres integration coverage through an explicit harness-managed command until the repo intentionally promotes it into the default or CI baseline.
- The first harness-managed verification command is `uv run wiki-agent-integration test`.
- The harness-owned `test` path should delegate to pytest integration cases rather than embedding primary scenario assertions inline in the harness script.
- Future real runtime integration coverage should be added as more pytest cases under `tests/integration/` and executed through the same harness-managed command path.
